# Routing

Follow me on Twitter, happy to take your suggestions on topics or improvements /Chris

When you build a server application, you need to consider a few things namely:

  • Static resources, will it have static resources, how do I serve those?
  • Query parameters, will I need to support query parameters?
  • Routing, will part of my application support API like routes that responds with JSON. Additionally what HTTP verbs will I support for my API resources?

If we support all of the above, we suddenly need to add a lot of code that is easy to maintain and extend.

Let's first learn how to tackle all the above things. Secondly let's discuss architecture.

# Static resources

A static resource is simply a file. Common types of files, when you build a web application is:

  • Images, our application will most likely need images
  • CSS, our application definitely needs styling
  • Other, this might be a range of files that you might want to let users access like reports in PFD, text files and more.

The common denominator with the above file types is that they will be accessed using their full URL, like so http://host/route/image.jpg.

It's easy to start building an application looking like this:

const http = require('http');
const path = require('path');
const fs = require('fs');

const server = http.createServer((req, res) => {
  // if part of the URL is static then treat the same
  if (req.url == '/static/test.txt') {
    const read = fs.createReadStream(path.join(__dirname , req.url));
    read.pipe(res);
  } else if (req.url == '/static/styling.css') {
    const read = fs.createReadStream(path.join(__dirname , req.url));
    read.pipe(res);
  } else {
    res.end('ok')
  }
})

server.listen(8000, () => {
  console.log('Server running on 8000')
});

This code can be improved by adding a Regular Expression. If all our static files are in a static directory, then let's test for that, like so:

const http = require('http');
const path = require('path');
const fs = require('fs');

const server = http.createServer((req, res) => {
  // if part of the URL is static then treat the same
  if (/\/static\//.test(req.url)) {
    const read = fs.createReadStream(path.join(__dirname , req.url));
    read.pipe(res);
  } else {
    res.end('ok')
  }
})

server.listen(8000, () => {
  console.log('Server running on 8000')
});

# Routes

We have two types of routes:

  1. A static file, this is when we ask for a specific resource by its full path and file name, like so /route/products.json
  2. An API like route, this is when we ask for a resource by name while following some kind of convention like REST, SOAP, GraphQL, or similar approaches. Here our route might look like this instead /route/products. The way we ask for all the data on a resource or specific data differs. We either encode that information in the route itself, like so /route/products/1 or using query parameters /route/products?id=1.

Let's focus on the latter version and see how we can solve this approach through code and by using our http module.

# Matching a URL

When we have a server application, we want to send the user to the correct place in our code. If the user is asking for /products, they are most likely asking for a list of products. If they are instead asking for /products/1, they are most likely asking for a specific product. This means that we need to handle these routes differently in code. You will need some kind of switching logic like so:

if (/*list like route*/) {
  // return a list
} else if (/*asking for one item*/) {
  // return one item
}

We can use RegEx to solve this. The method test() is made for this purpose. Let's first construct a RegEx that solves our list case e.g /products:

/\/products\/?/.test('/products') // true
/\/products\/?/.test('/products/') // true
1
2

Now let's match routes with a parameter in it, like this one /products/1:

/\/products\/\d+\/?/.test('/products/11') // true
/\/products\/\d+\/?/.test('/products/11/') // true
/\/products\/\d+\/?/.test('/products/aab') // false
1
2
3

If your IDs are let's say GUIDs over integers, you might need a different pattern than \d+. In that case, we need something like \w+{4}-\w+{4}-\w+{4}-\w+{4}.

Our updated switching logic will now look like so:

const http = require('http');
const server = http.createServer(() => {
  const products = [{
    id: 1, name: 'product'
  },
  {
    id: 2, name: 'product2'
  }];

  if (/\/products\/?/.test(req.url)) {
    res.end(products);
  } else if (/\/products\/\d+\/?/.test(req.url)) {
    const { groups: { productId } } = req.url.match(/\/products\/(?<productId>\d+)/);
    res.end(products.find(p => p.id == productId))
  }
})

server.listen(8000, () => {
  console.log('Running on port 8000')
})

# Parsing a URL

We've learned above how we can match routes of different shapes and return different results because of it. Another interesting thing we might want to do is to break apart our URL in different pieces. Let's have a look at a URL and see what's what:

http://localhost:8000/products?page=1&pageSize=20
1

Lets list what the above URL consists of:

  • http, this part is called a protocol
  • localhost, this part is called the host
  • 8000, this part is the port
  • /products?page=1&pageSize=20, this is the route, the last part of our URL. This can also include query parameters that help us filter the resource we are asking for.

Let's write a RegEx capable of extracting the above information:

const { groups } = 'http://localhost:8000/products?page=1&pageSize=20'.match(/(?<protocol>\w+):\/{2}(?<hostname>\w+):(?<port>\d+)(?<route>\/.*)/)
// groups = { protocol: 'http', hostname: 'localhost', port: 8000, route: '/products?page=1&pageSize=20' }
1
2

It should be said though that the http module is able to do much of this parsing already. To construct the full URL, you need something like this:

var rawUrl = /^http:/.test(req.url) ? req.url : 'http://' + req.headers.host + req.url

# Query parameters

What is a query parameter? You might have seen routes like this http://host/route?page=1&pageSize=20. You recognize query parameters as everything happening after the ?. They are key-value pairs separated by &. They are used to filter a resource but can be used for other purposes. In the above example, we have the query parameters page and pageSize. In this case, they are meant to filter a resource by asking for a specific page from a long list of records. Additionally we also want to define the number of records on a page. This is great if we have a resource consists of hundreds or millions of records. With that much data, it simply isn't feasible to return all the records. By using query parameters, we can limit how much data is returned back.

So how can we parse out the query parameters? Well, there are two ways:

  1. Use the built-in url module
  2. Use a RegEx to get what we need

Let's check the built-in url module first.

const http = require('http');
const url = require('url');

http.createServer((req, res) => {

  // req.url /products?page=1&pageSize=20
  // queryParams: { page: 1, pageSize: 20 }

   const queryParams = url.parse(req.url, true)  
})

The other way is just two lines of code, so it's up to you if you want to pull in the url module for solving this problem. Here is the solution:

http.createServer((req, res) => {

  // 0) Assume, req.url = /products?page=1&pageSize=20
  // 1) RegEx result:  ['page=1', 'pageSize=20']
  // 2) queryParams: { page: 1, pageSize: 20 }

  // 1)
  const pairs = req.url.match(/(\w+\=\w+)/g);

  // 2)
  const queryParams= pairs.reduce((acc, curr) =>, {
    const [key, value] = curr.split(':');
    return { ...acc, [key]: value };
  }, {})
})
